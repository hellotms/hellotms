## 2. How to Sign & Upload New Versions

### Step 1: Set environment variables
```powershell
$env:TAURI_SIGNING_PRIVATE_KEY = (Get-Content "apps/admin/admin_tauri.key" -Raw).Trim()
$env:TAURI_SIGNING_PRIVATE_KEY_PASSWORD = "TMS2024"
```

### Step 2: Build with auto-signing
```bash
pnpm tauri build --bundles nsis,msi
```
Tauri CLI will AUTOMATICALLY sign the generated installers using the environment variables above.
The `.sig` files will appear next to the `.exe` and `.msi` files.

### Step 3: Upload to Admin Panel
1. Upload the generated EXE/MSI binary.
2. **CRITICAL:** The `.sig` file generated by Tauri is **base64-encoded**. You MUST:
   - Open the `.sig` file
   - **Decode the base64 content** (it will resolve to 4 lines of text)
   - **PASTE THE DECODED 4-LINE CONTENT** into the "Update Signature" field
3. Tauri v2 requires the full Minisign format:
   ```
   untrusted comment: signature from tauri secret key
   RWRBM...AQI=
   trusted comment: timestamp:XXXXXXXXXX  file:tms-Portal_X.X.X_x64-setup.exe
   D55paM...Bw==
   ```

### Simple One-Command Build + Sign
```bash
pnpm run build:admin:signed
```
This automatically loads the private key from `admin_tauri.key` and signs the installer. No manual env setup needed.

## 3. Critical Rules

> [!CAUTION]
> ### NEVER DO THIS:
> 1. **NEVER use a custom Node.js signing script** (`sign_pure_node.cjs`). Always use `npx tauri signer sign` or build-time auto-signing.
> 2. **NEVER paste the Public Key (`RWRB...` or `RWSF...`) in the Signature field**. The signature and public key are DIFFERENT things.
> 3. **NEVER generate keys with tools other than `npx tauri signer generate`**. Other tools may produce incompatible key formats.

> [!WARNING]
> ### Key File Format:
> - The `.key` file generated by `npx tauri signer generate` is **base64-encoded text** (not raw binary).
> - When decoded, it contains: `untrusted comment: rsign encrypted secret key\n<base64 key data>`
> - The key data is in standard **minisign encrypted format** (154+ bytes).
> - To pass to `TAURI_SIGNING_PRIVATE_KEY` env var: use the **raw file content** (it's already base64).

## 4. Troubleshooting

### Error: "The signature ... could not be decoded"
**Root Cause Checklist:**

| # | Check | How to verify |
|---|-------|---------------|
| 1 | Is the `.sig` content in the correct 4-line minisign format? | Open `.sig`, decode base64, verify 4 lines starting with `untrusted comment:` |
| 2 | Is the public key in `tauri.conf.json` correct? | Must match the `.key.pub` file |
| 3 | Was the signature generated by Tauri CLI (not a custom script)? | Check that Line 2 of decoded `.sig` is exactly 100 chars (74 bytes decoded) |
| 4 | Do the key ID in the signature match the public key? | Both should reference the same Key ID |

### Error: "incorrect updater private key password: failed to fill whole buffer"
**The key file is in an incompatible format.** This happens when:
- Key was generated by an old tool or different minisign implementation
- Key file size is NOT ~154+ bytes when base64-decoded
- **Solution:** Regenerate keys using `npx tauri signer generate -p "PASSWORD" -w path/to/key`

### Error: "failed to decode base64 secret key: Invalid symbol X, offset Y"
**The key has formatting issues.** Check for:
- Extra whitespace or newlines in the key file
- Windows CRLF (`\r\n`) instead of LF (`\n`)
- The `.key` file should be a single base64 blob (no line breaks in the base64 content)

### Error: "Update failed: undefined"
- The version on the server is the SAME as the installed app. Updater only triggers when `ServerVersion > LocalVersion`.

## 6. Lessons Learned (Post-Mortem)

> [!IMPORTANT]
> ### Key Lessons from the v0.1.2 — v0.1.8 Signing Crisis:
> 
> 1. **ALWAYS use Tauri's official CLI for signing.** Custom scripts that manipulate raw cryptographic bytes are fragile and error-prone.
>
> 2. **Minisign key formats vary.** Different tools (minisign, rsign, rsign2, Tauri CLI) may produce keys in slightly different binary formats. Only use keys generated by `npx tauri signer generate`.
>
> 3. **Tauri's error messages are misleading.** When signature verification fails, Tauri sometimes shows the PUBLIC KEY in the error message (e.g., "The signature RWSF/... could not be decoded"). This makes it look like the pubkey is the problem, but the real issue is the SIGNATURE.
>
> 4. **Never extract raw bytes from key files manually.** Ed25519 keys have specific encoding requirements (seed vs expanded key, PKCS#8 format, algorithm prefixes). Getting any of these wrong silently produces invalid but seemingly-valid signatures.
>
> 5. **The `.sig` file in Tauri v2 is base64-encoded.** The raw file content is a base64 string. When decoded, it reveals the 4-line minisign format. The 4-line content (not the outer base64) is what goes into the database/admin panel.
>
> 6. **Test signing BEFORE deploying.** Use `npx tauri signer sign` to manually sign a test file and verify it works before building the production installer.
>
> 7. **When server returns correct data but app still fails:** Check if the installed app was built with a MATCHING public key. Key mismatches can't be fixed server-side.

> [!CAUTION]
> If you ever lose `admin_tauri.key` OR the password, you will lose the ability to auto-update your current users. Keep a secure backup of this file outside of the project.
